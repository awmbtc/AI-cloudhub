# AI-cloudhub 风险与成本评估（相对架构定稿）

> 依据 [ARCHITECTURE.md](./ARCHITECTURE.md) 最强解评估。  
> 金额为 **人民币量级估算**（2026），非报价；人力按「能写 Go 的全栈/后端」人月粗算。  
> **决策基线与黑名单的正式记录：** [DECISIONS.md](./DECISIONS.md)（D-001）。

---

## 1. 总览

| 维度 | 评级 | 一句话 |
|------|------|--------|
| **方向风险** | 低 | BYOS + 路径无感 + Agent 契约，避开自建存储与网盘价格战 |
| **交付风险** | **中高** | 价值集中在 **Runtime 自动挂载**（hubd/FUSE/Windows），工程难度高于 CRUD API |
| **规模成本（我们）** | **低** | 控制面可轻；对象流量在用户侧 |
| **规模成本（用户）** | 视用量 | R2/OSS 账单随存量和下载变 |
| **安全风险** | **高（若做错）** | 代管 API Key；STS/加密/吊销做不好会出事 |
| **综合** | 值得做 | P0 必须做对 Runtime+凭证；否则产品只是「rclone 配置生成器」 |

## 2. 风险清单

### 2.1 技术风险

| 风险 | 等级 | 影响 | 缓解 |
|------|------|------|------|
| **FUSE/WinFsp 稳定性** | 高 | zip/Office/杀软导致卡死、句柄锁、挂载掉线 | 官方 Runtime 统一版本；写缓存档位；write barrier；已知软件黑白名单 |
| **rclone 依赖过重** | 中 | 升级/行为变更、授权（MIT 可用） | `mountlib` 抽象；可替换后端；钉版本 |
| **「无感」预期过高** | 高 | 用户拿网盘对比 Dropbox 原生同步，嫌慢/怪 | 产品明确 mode（mount vs sync）；缓存盘要求 SSD；不承诺比特级等于本地盘 |
| **云端无 FUSE 环境** | 中 | Serverless/部分容器禁 FUSE | 提供 `sync_workspace` / `direct` 模式；Runner 镜像自带 FUSE 权限说明 |
| **多端同时写冲突** | 中 | 本机+云端同写一文件损坏 | 文档约定；可选文件锁/冲突副本；Agent 单写者优先 |
| **厂商 API 差异** | 中 | OSS/COS/七牛边界 case | 批次推进；兼容性测试矩阵；S3 兼容优先 |
| **自研同步引擎膨胀** | 中 | 若过早做 Seafile 级同步，工期爆炸 | P0–P2 **禁止**自研完整同步协议；先 mount+缓存 |
| **遗留 workspace 代码** | 低 | 口径混乱 | 标记 deprecated，默认关闭 |

### 2.2 安全与合规风险

| 风险 | 等级 | 影响 | 缓解 |
|------|------|------|------|
| **代管用户云密钥** | **高** | 泄露 = 用户桶被拖 | 信封加密、KMS、最小权限 STS、审计、不落盘永久 Key |
| **长期 Key 进 conf/镜像** | **高** | 扩散难收回 | 架构强制 STS；扫描禁止 secret 入镜像 |
| **恶意租户当代理刷流量** | 中 | 账单/信誉 | 鉴权、限流、设备绑定、异常检测 |
| **数据出境（R2 等）** | 中（国内客户） | 合规顾虑 | 厂商可选国内 OSS/COS；协议披露 |
| **开源 AGPL 污染** | 低 | 若错误链入 AGPL 服务端库进发行 | 控制面用 Apache/MIT 依赖；MinIO **服务**只作用户后端不打包进我们发行核心 |
| **审计不足** | 中 | 出事无法追责 | mount/STS/登录审计落库 |

### 2.3 产品与市场风险

| 风险 | 等级 | 影响 | 缓解 |
|------|------|------|------|
| **「只是 rclone 套壳」认知** | 高 | 无付费意愿 | 强调 Agent 契约、云 Job、多用户策略、自动挂载闭环 |
| **Agent 生态碎片** | 中 | 各家 Agent 不认 env | Manifest + MCP + 一键规则；先吃透 1～2 个主流 Agent |
| **竞品** | 中 | rclone GUI、云盘同步盘、AList | 差异化：Agent 一等 + 云本机统一 + 多 Key 盘体系 |
| **命名 AI-cloudhub 宽泛** | 低 | SEO/心智不清 | 副标题固定「Agent Workspace Disk OS」 |

### 2.4 运营与交付风险

| 风险 | 等级 | 影响 | 缓解 |
|------|------|------|------|
| **P0 延期（hubd）** | 高 | 长期停留在 demo | 砍范围：先 Win 或先 macOS 一条线；Linux Runner 并行 |
| **Windows 适配耗时** | 高 | 国内桌面主力 | 单列 Windows 里程碑；WinFsp 安装体验 |
| **客服成本** | 中 | 挂载问题多 | 健康检查、一键诊断日志、状态机 UI |
| **人力不足却铺 Job 平台** | 中 | 两边不靠谱 | 严格 P0→P1→P2；Job 放 P3 |

### 2.5 风险矩阵（摘要）

```text
        影响大
          │  代管 Key 泄露     FUSE/无感翻车
          │  永久 Key 落盘     hubd 做不出来
          │
          │  多端冲突          厂商差异
          │  限流不足
          │
          │  命名/SEO          遗留代码
          └──────────────────────── 发生概率 →
```

**最高优先缓解：密钥体系（STS+加密）+ Runtime 稳定性 + 预期管理。**

## 3. 成本评估

### 3.1 研发成本（做到架构定稿可用）

按 **1 名熟练 Go + 0.5 前端/客户端** 粗算人月（可折算成团队）：

| 阶段 | 范围 | 人月（约） | 说明 |
|------|------|------------|------|
| **已投入** | API 骨架、Provider A、Drive mount 下发 | 0.5～1 | 已完成量级 |
| **P0** | hubd 自动挂载、STS、Manifest、runner 统一 mountlib | **2～4** | 最大不确定性：桌面挂载 |
| **P1** | PG、密钥加密、Binding 状态、限流、审计 | **1.5～2.5** | 常规后端 |
| **P2** | 缓存档位、write barrier、区域策略、厂商 B | **1.5～3** | 含兼容测试 |
| **P3** | Job 平台、MCP、可观测、厂商 C | **2～4** | 可裁剪 |
| **合计到「能卖/能用」P0+P1** | | **约 4～7.5 人月** | 建议最小闭环 |
| **合计到较完整 P0–P2** | | **约 6～11 人月** | 性能与多云像样 |
| **P0–P3 全量** | | **约 8～15 人月** | 平台化 |

若按 **市场人力成本 3～5 万/人月**（含社保，量级）：

| 目标 | 研发费用量级 |
|------|----------------|
| 最小可演示增强（半吊子 P0） | 5～15 万 |
| **P0+P1 可内测/小规模** | **15～35 万** |
| P0–P2 较完整 | 25～50 万 |
| 全量 P3 | 35～70 万+ |

一人全职可拉长到 6～12 个月日历时间（P0+P1）。

### 3.2 基础设施成本（我们侧 · 月）

架构定稿下 **我们几乎不为用户文件付费**：

| 阶段 | 部署 | 元/月（约） |
|------|------|-------------|
| 开发 | 笔记本 + 可选轻量 | 0～50 |
| 内测 | 轻量 2C4G 年付摊销 + 域名 | **20～80** |
| 公测多用户 | API×1～2 + 托管/同机 PG Redis | **100～500** |
| 正式（万级注册元数据） | API 多副本 + PG + Redis + LB + 日志 | **500～3,000** |
| 自建大规模 Runner 池 | 算力另计 | **按并发 Agent 线性**（可 BYOC 转嫁用户） |

**对象存储：** 用户 R2/OSS 账单；我们 **0**（除非自营默认桶，架构不推荐）。

### 3.3 用户侧成本（转嫁，需产品诚实）

| 用户选择 | 体感成本 |
|----------|----------|
| R2 免费档 10GB | ≈ 0 |
| R2 1TB 级 | 约百元/月量级存储（出口 0） |
| 国内 OSS + 公网狂下 | 流量可能远超存储 |
| 自建 MinIO 轻量盘 | 年付服务器摊销 |

**风险：** 用户低估流量费 → 归因产品；文档与配额提示必须做。

### 3.4 机会成本 / 不做最强解的成本

| 停留在 MVP | 后果 |
|------------|------|
| 只做 conf 生成器 | 无壁垒、难收费、被 rclone 替代 |
| 不做 STS | 安全事故一票否决 |
| 过早自建存储 | 成本与架构定稿背离，回到「贵」 |

## 4. 分阶段「风险-成本」曲线

```text
成本/风险
  │
  │                 ╱── P3 平台化（可选，成本升、风险可控）
  │            ╱────
  │       ╱── P2 性能多云（中成本）
  │  ╱── P1 多用户安全（中成本，降安全风险）
  │╱ P0 Runtime（成本中高，降产品风险最大）
  │ MVP 骨架（低成本，高产品风险：像套壳）
  └────────────────────────────► 时间
```

| 阶段 | 我们现金成本 | 主要剩余风险 |
|------|--------------|--------------|
| 现在 MVP | 极低 | 无感/安全/认知 |
| P0 完成 | 研发为主 | FUSE 边界 case |
| P1 完成 | 小额云资源 | 运营与客服 |
| P2 完成 | 中等研发 | 厂商长尾 |
| 自建 Runner 大池 | **云主机成为主成本** | 与「穷部署」冲突 → 应用 BYOC |

## 6. 决策建议（定稿 · 必须遵守）

> 本节为 **产品/工程决策基线**。与 [ARCHITECTURE.md](./ARCHITECTURE.md) 冲突时，以「安全与成本红线」为准修订实现，不悄悄上马黑名单项。

### 6.1 必须坚持（绿灯）

| 决策 | 理由 |
|------|------|
| **BYOS**，用户自带对象存储 | 我们侧成本不随文件量爆炸 |
| **控制面极瘦**，字节不中转 | 可水平扩展、结构健康 |
| **本机 hubd + 云 runner 同一套 mount/契约** | 差异化；双执行面一等公民 |
| **Go 自研**，不魔改 MinIO、不自研对象存储内核 | 避开许可/运维深水区 |
| **落地顺序 P0→P1→P2→P3** | 先无感与安全，再性能与平台 |

### 6.2 必须买入的「保险」（不做 = 风险不可接受）

| # | 决策 | 挡什么 |
|---|------|--------|
| 1 | **STS + Key 信封加密** | 密钥事故、长期 Key 落盘 |
| 2 | **hubd / runner 自动挂载**（L2 一等产品） | 停在「rclone 套壳」 |
| 3 | **Manifest + `AI_CLOUDHUB_WORKSPACE` 等 env** | Agent 乱写路径 |
| 4 | **预期管理**（mount ≠ 本地 NVMe；写明 mode） | 体感差评与客诉 |
| 5 | **write barrier**（任务/关闭后刷缓存） | 假完成、跨端看不到 |

### 6.3 最小可行集合（推荐资源上限）

```text
最小「架构不丢人」= P0 + P1 安全子集
  · 人月：约 4～7.5
  · 控制面云资源：< 500 元/月（内测～早期公测）
  · 不做：P3 大平台、自研同步引擎、厂商一次做完、自建大规模 Runner 池
```

### 6.4 可砍 / 延期（黄灯 · 控成本）

| 项 | 省下 | 条件 |
|----|------|------|
| P3 托管 Job 大平台 | 2～4 人月 | 用户自带机跑 runner 镜像即可 |
| 自研完整块同步引擎 | 可能 6+ 人月 | 永久用 mount+缓存，勿第一年重写 Seafile |
| 厂商 C 批一次做完 | 测试与维护 | 按 VENDORS 批次来 |
| 豪华可观测/多活 | 云与人力 | P1 先有基础日志与审计 |

### 6.5 黑名单（红灯 · 默认禁止）

下列项 **默认不做**；若业务强需求，须单独评审并改成本模型，不得当成「架构定稿的默认路径」。

| 黑名单项 | 为何拉黑 |
|----------|----------|
| **自建大规模 Runner 池**（见 §6.6） | 把我们重新变成「算力账单公司」，毁掉 BYOS 成本优势 |
| 魔改 / fork MinIO 当产品内核 | 口径、许可、升级全错 |
| 控制面默认中转文件 body | 性能与成本双杀 |
| 平台替用户存全站对象数据 | 回到网盘成本曲线 |
| 过早自研完整同步协议 | 交付风险与人月失控 |
| 多租户共享同一 `/workspace` 挂载 | 安全事故 |

### 6.6 黑名单详解：禁止「自建大规模 Runner 池」

#### 它是什么

```text
「自建大规模 Runner 池」=
  我们自己买/租大量云主机或 K8s 节点，
  常驻或弹性拉起成百上千个 Agent 容器，
  替（几乎）所有用户跑云端 Agent，
  算力与带宽账单主要由我们承担。
```

典型错误 enticement：

- 「有云端 Agent 需求，所以我们要自建 500 节点 Runner 集群」  
- 「对标云 IDE，算力我们包了」  
- 架构图里把 **runner 池** 画成默认、唯一的云执行方式，且不写 BYOC  

#### 为什么是黑名单（不是「永远禁止云端 Agent」）

| 维度 | 自建大规模池 | 架构定稿允许的做法 |
|------|--------------|-------------------|
| **成本归属** | CPU/GPU/磁盘/公网随任务线性涨，**我们付** | **用户自带机 / 用户云账号** 跑官方 runner 镜像；或用户按量用**第三方**算力 |
| **与 BYOS 一致性** | 存储 BYOS 了，算力却「我们全包」→ 半吊子 | 存储 BYOS + **计算 BYOC（Bring Your Own Compute）** |
| **和穷部署冲突** | 控制面月费几十～几千，Runner 池月费可轻松 **数万～数十万** | 控制面保持轻；算力不进我们 P&L 主路径 |
| **运维** | 节点、镜像、抢占、脏盘、多租隔离、安全逃逸 | 我们只交付 **镜像 + 契约 + 调度 API（可选）** |
| **安全面** | 多租户同集群 = 更大攻击面与噪音邻居 | 用户自有账号隔离最干净 |
| **产品焦点** | 沦为「卖云主机的 Agent」 | 聚焦 **盘映射 + 自动挂载 + 契约** |

**云端 Agent 仍然要做**，但形态必须是：

```text
✅ 允许（白名单）
  1. 官方 runner 镜像 / cloud-agent-entry
     → 用户在自己的 ECS/轻量/K8s/GitHub Actions/自有机运行
  2. 本机 hubd（用户自己的电脑算力）
  3. 可选：极小规模「演示/内测」Runner（个位数节点，有预算帽）
  4. 可选：对接用户云的弹性（用用户的 AK 在用户账号里 CreateInstance）
     → 账单进用户云账号，不是我们主账户无脑扩容

❌ 禁止（黑名单默认）
  · 在我们云账号下维持大规模常驻/无上限弹性 Agent 农场
  · 以「免费无限云端 Agent」为卖点却无算力计量转嫁
  · 未做 STS/隔离就多租户混部大池
```

#### 若未来商业上「必须」我们出算力

不得偷偷做，须同时满足：

1. **单独产品线/SKU**（算力包），与 Drive Map 核心订阅拆开定价  
2. **硬配额 + 计量计费**（超时杀、并发帽、磁盘帽）  
3. **预算熔断**（账户日/月上限）  
4. **成本模型过会**（修改 RISK-COST 与定价表）  
5. 架构上仍优先 **用户云账号执行**（STS 扮演用户侧）  

未满足以上 5 条 → **维持黑名单**。

#### 和架构文案的对齐

| 说法 | 正确理解 |
|------|----------|
| 「Cloud Runtime / runner 镜像」 | 软件交付物，**跑在哪由用户或按量 SKU 决定** |
| 「Runner 与存储同区域」 | **调度建议**，不是「我们自建全球机房」 |
| 「可扩缩 runner 池」 | 指 **用户侧或计费 SKU 池**，默认 **不是** 我们免费大池 |

## 7. 量化摘要（给老板/自己）

| 问题 | 答案 |
|------|------|
| 最大风险是什么？ | ① 密钥事故 ② 挂载体验翻车 ③ 停在套壳无 Runtime |
| 最大成本在哪？ | **研发人月**，不是控制面服务器 |
| 做到能内测大概多少钱？ | **研发 15～35 万量级** 或 **1 人 6～9 个月** + 云 **几十～几百元/月** |
| 用户多了我们会不会爆？ | 控制面不易爆；**若自建大规模 Runner 池则会爆** → 故列黑名单，算力 BYOC |
| 和「自建网盘存 10 万用户文件」比？ | **我们侧固定成本低一个数量级以上**；风险从「磁盘账单」变为「客户端与密钥」 |
| 最小决策集？ | **P0+P1 + 四条保险 + 黑名单（含禁止大规模 Runner 池）** |

---

## 8. 一句话

> **最强解成本结构健康（我们侧轻），交付风险在 Runtime 与密钥；用 4～7 人月拿下 P0+P1 是最佳对冲。**  
> **云端 Agent 要做，但禁止默认「自建大规模 Runner 池」把公司做成算力冤大头——算力 BYOC，我们只交付镜像、契约与控制面。**
